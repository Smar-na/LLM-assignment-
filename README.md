# Toxic Comment Classification using LLM Prompt Engineering

Multi-label toxic comment classification on the [Jigsaw Toxic Comment Classification dataset](https://www.kaggle.com/competitions/jigsaw-toxic-comment-classification-challenge/data) using GPT-4.1 prompt engineering pipelines and BERT-based baselines. Six toxicity labels are classified: `toxic`, `severe_toxic`, `obscene`, `threat`, `insult`, `identity_hate`.

## Data

### Download the Jigsaw dataset

Download `train.csv` from Kaggle:

```bash
kaggle competitions download -c jigsaw-toxic-comment-classification-challenge
unzip jigsaw-toxic-comment-classification-challenge.zip
```

Or download manually from: https://www.kaggle.com/competitions/jigsaw-toxic-comment-classification-challenge/data

Place `train.csv` in the `data/` directory.

### Generate the evaluation and few-shot pool files

Run `EDA_And_Data_Sampling.ipynb` end to end to produce the evaluation set and few-shot pool. This notebook:

- Reserves examples per rare class (`threat`, `identity_hate`, `severe_toxic`) for the few-shot pool, strictly excluded from the evaluation set (zero leakage)
- Produces a stress-test evaluation set with all available rare-class comments plus matched clean comments (approximately 8,000 rows, 50/50 toxic/clean split), which is the definitive benchmark used consistently across all models and BERT baselines
- Excludes manually specified comment IDs from the RAG pool
- Applies text cleaning: drops null comments, handles encoding errors, and strips control characters
- Saves `stress_test_eval_set.csv` and `few_shot_pool_fixed_3.csv`
- Generates EDA visualisations of class distribution and label correlations

> **Note:** `EDA_And_Data_Sampling.ipynb` is a Jupyter notebook, not a Python script. Run all cells in order rather than executing it from the command line.

## Running the Pipelines

All notebooks are self-contained and designed for Google Colab or a local Jupyter environment. Each notebook includes checkpoint/resume support, so if a run is interrupted, re-running it will automatically skip completed rows and continue from where it left off.

### BERT Baselines: `BERT.ipynb`

**Required files:** `stress_test_eval_set.csv`

> **Note:** The notebook code also attempts to load `stratified_evaluation_set_8k.csv`, which is not included in this repository. This load can be ignored or commented out. All results reported in the paper and discussed in this README use `stress_test_eval_set.csv` only, which is consistent with Methods 1 to 4.

Run all cells in order. The notebook will:

1. Load `stress_test_eval_set.csv`
2. Run inference with `google-bert/bert-base-cased` (randomly initialised 6-label classification head, `num_labels=6`, sigmoid activation, threshold=0.5), which serves as an uninstructed lower-bound baseline. The load warning `classifier.weight MISSING` is expected and confirms the head is randomly initialised
3. Run inference with `unitary/toxic-bert` (domain-adapted model fine-tuned on Jigsaw, threshold=0.5), which is the primary supervised BERT benchmark
4. Output per-label Precision, Recall, F1, and Accuracy reports for both models via `classification_report`

Both models use the Hugging Face `pipeline` with `function_to_apply="sigmoid"` and truncation at 512 tokens. A fixed threshold of 0.5 is applied uniformly across all six labels with no per-label tuning.

### Method 1: Zero-Shot — `METHOD1_zero_shot_.ipynb`

**Required files:** `stress_test_eval_set.csv`

**Output:** `predictions_zero_shot.csv`

Run all cells in order. The notebook will:

1. Load the evaluation set
2. Run async GPT-4.1 inference (`ASYNC_CONCURRENCY=8`, `temperature=0`, `seed=42`)
3. Rescale binary 0/1 model outputs to 0/100 and apply per-label thresholds to produce final binary predictions
4. Save predictions with confidence scores, token counts, and per-row runtime
5. Print a full evaluation report: per-label F1, ROC-AUC, Hamming Loss, and cost breakdown

### Method 2: Few-Shot Static — `METHOD2_few_shot_simple.ipynb`

**Required files:** `stress_test_eval_set.csv`

**Output:** `predictions_few_shot41.csv`

Run all cells in order. The notebook will:

1. Load the evaluation set
2. Inject 41 static few-shot examples covering all 41 observed label combinations in the Jigsaw dataset into every prompt (drawn from `few_shot_pool_fixed_3.csv`)
3. Run async GPT-4.1 inference
4. Save predictions and print full evaluation report

### Method 3: RAG Dynamic — `METHOD3_RAGDYNAMIC.ipynb`

**Required files:** `stress_test_eval_set.csv`, `few_shot_pool_fixed_3.csv`

**Output:** `predictions_rag_dynamic.csv`, `pool_embeddings.npz` (auto-generated and cached)

Run all cells in order. The notebook will:

1. Load the evaluation set and few-shot pool
2. Embed the full pool using `text-embedding-3-small`, cached to `pool_embeddings.npz` after the first run, with subsequent runs loading from cache automatically
3. For each test comment, retrieve the top-7 most semantically similar pool examples via cosine similarity (similarity floor: 0.10), with at least 3 of the 7 slots reserved for hard-minority-label positives (`severe_toxic`, `threat`, `identity_hate`), implemented via `RAG_K=7` and `RAG_HARD_K=3` in the config
4. Run async GPT-4.1 inference with dynamically retrieved examples per comment
5. Save predictions and print full evaluation report including embedding cost breakdown

### Method 4: HA-RAG — `METHOD_4_RAG_HybridAugmentation.ipynb`

**Required files:** `stress_test_eval_set.csv`, `few_shot_pool_fixed_3.csv`

**Output:** See Output Files Reference table below

This is the most sophisticated pipeline. Run the workflow functions in order in the notebook:

```python
# Step 1: Split evaluation set into dev and held-out test (run once)
split_eval_set()

# Step 2: Build label-aware embedding index (run once)
# Also generates curated_hard_labels_cl1.json for optional manual review
build_index()

# [Optional] Step 2b: Manual curation
# Open curated_hard_labels_cl1.json, set "keep": true for high-quality
# hard-label examples, then re-run build_index() to incorporate them

# Step 3: Classify the dev set
run_inference(split="dev")

# Step 4: Tune per-label thresholds on dev set with recall floors
best_thresholds = tune_thresholds_on_dev()

# Step 5: Load tuned thresholds and classify held-out test set
best_thresholds = load_dev_thresholds()
run_inference(split="test", thresholds=best_thresholds)

# Step 6: Final evaluation on held-out test set
evaluate_predictions(split="test")
```

> **Note on async:** The notebook defines both `run_inference_async()` (async) and `run_inference()` (synchronous wrapper). The workflow above uses `run_inference()` which handles the async internals automatically and is the recommended entry point for standard Jupyter or Colab environments.

> **Note on runtime:** `build_index()` runs once and takes approximately 20 to 25 minutes depending on hardware. Subsequent runs load the saved index from `label_aware_index_v5_cl1.npz` automatically. Inference on the test split (4,001 rows) takes approximately 18 to 20 minutes at `ASYNC_CONCURRENCY=8`.

Key features of Method 4:

- 4-point severity rubric with dedicated second-pass severity check for borderline `severe_toxic` cases (trigger: `toxic >= 40` AND `20 <= severe_toxic <= 65`)
- Threat credibility filter with boundary contrast examples (IS / IS NOT a threat) distinguishing rhetorical anger from actionable language
- Group-noun detection heuristic with dedicated second-pass identity check for borderline `identity_hate` cases (trigger: hint >= 0.08 AND `15 <= identity_hate <= 55`)
- Skip-gate for high-confidence clean comments, bypassing 69 of 4,001 test comments (approximately 1.7%) and saving approximately $0.13
- Post-processing heuristic rules enforcing label hierarchy consistency (R1 to R5, applied in order: R4, R1, R2, R3, R5)
- Adaptive self-consistency voting for hard labels only (`severe_toxic`, `threat`, `identity_hate`), triggered for approximately 25% of comments
- Label-aware retrieval using `RETRIEVE_K=6` by default and `RETRIEVE_K_HIGH=8` when hard-label hints are detected
- Per-label threshold tuning on dev set with recall floors: `severe_toxic` and `threat` >= 0.50, `identity_hate` >= 0.60

## Important Notes

**API keys:** Never commit API keys to the repository. All notebooks use `os.getenv("OPENAI_API_KEY")` and the key should be set as an environment variable only.

**Cost:** Running all four GPT-4.1 pipelines on the full evaluation set incurs API costs. Method 2 is the most expensive ($24.09) because every prompt carries the full 41-example block (approximately 28.6M input tokens total). Method 4 ($7.82 on the 4,001-row test split) costs less than Method 3 ($11.59 on 8,000 rows) because dynamically retrieved prompts are shorter than the static 41-example block, despite multi-pass inference for borderline cases. Note that Method 4's cost covers only 4,001 rows while Methods 1 to 3 cover 8,000 rows, so the per-sample costs are more comparable than the totals suggest.

**Runtime:** Methods 1 to 3 complete in approximately 18 to 22 minutes on the full 8,000-row evaluation set at `ASYNC_CONCURRENCY=8` on a paid OpenAI tier (M1: approximately 20 minutes, M2: approximately 18 minutes, M3: approximately 22 minutes). Method 4 inference on the 4,001-row test split takes approximately 18 to 20 minutes; `build_index()` runs once and takes an additional 20 to 25 minutes on first execution.

**Evaluation set design:** The stress-test evaluation set deliberately over-represents rare classes relative to the natural Jigsaw distribution. Macro-F1 scores are therefore not directly comparable to published Jigsaw leaderboard results.

**Claude Haiku 3:** A Claude Haiku 3 pipeline was tested on a 1,000-row subset; results are reported in the Appendix of the report. Full-scale evaluation was not feasible due to free-tier API rate limits restricting concurrency to one request at a time.
